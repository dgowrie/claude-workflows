Memory System

How Claude carries context between sessions

How Memory Works

  • Markdown files with frontmatter in projects/<path>/memory/
  • MEMORY.md is the index — one-line summaries loaded every session
  • Individual files read only when they seem relevant
  • Scoped per project directory — invisible across projects

Memory Types

TypePurposeExample
feedback Behavioral corrections "No Co-Authored-By unless fully agentic"
project Context not in the code "React 19 upgrade follows sibling plugin pattern"
user Who you are, what you know "Deep Go expertise, new to React"
reference Pointers to external systems "Bugs tracked in Linear project INGEST"

Memory Limitations

ConcernDetail
ScopingProject-directory-scoped. Global lessons need CLAUDE.md
StalenessNo expiry or auto-cleanup
ConflictsCan contradict CLAUDE.md or other memories
TruncationIndex truncates past 200 lines
Best-effortNo guarantee Claude reads or writes them

Memory Maintenance

  • Prune — delete memories for shipped features and resolved bugs
  • Promote — move recurring feedback to CLAUDE.md rules
  • Seed — write memories manually when Claude keeps rediscovering things
  • Correct — fix bad memories directly, faster than hoping Claude self-corrects

Plans

Working documents for complex tasks

How Plans Work

  • Markdown files in plans/ with generated names
    e.g., clever-prancing-goblet.md
  • Created when Claude enters plan mode during a session
  • Claude writes the plan, you approve, Claude executes
  • Not auto-reloaded across sessions — persist on disk but must be explicitly referenced

What Plans Capture

  • Implementation blueprints — step-by-step, file-level changes
  • Situational snapshots — current state when things go sideways
  • Decision rationale — why an approach was chosen
    Often not in code or commits — only lives in the plan

Practical Use of Plans

  • Resume — "Read plans/toasty-gathering-elephant.md and pick up where we left off"
  • Review — revisit decision rationale after the fact
  • Pre-write — author a plan yourself and tell Claude to execute it
  • Clean up — delete completed plans; promote durable rationale to memories

Self-Correction Loop

Learning from mistakes across sessions

The Problem

  • Each session starts completely fresh
  • No recollection of past corrections unless written to disk
  • In practice, corrections get lost:
    • Claude prioritizes continuing the task
    • Treats correction as session-local, not generalizable
    • Session ends before circling back

Two Persistence Layers

LayerScopeDurabilityReliability
CLAUDE.md rules Global or per-project Permanent High
Feedback memories Per project directory Until pruned Medium

Rule of thumb: if it matters everywhere, it goes in CLAUDE.md.
If it's project-specific context, it's a memory.

Making It Reliable

  1. Mandatory, immediate edits — CLAUDE.md instruction requires stopping the task to propose a rule edit when corrected
  2. Explicit prompting — say "add this to CLAUDE.md" to force the edit
  3. Periodic review — run sessions to promote feedback memories to rules

Further improvements

  • Custom /correct skill
  • Session-end hook for correction review
  • Systematic memory-to-CLAUDE.md promotion

Key Takeaways

1. CLAUDE.md is the only reliable persistence. Always loaded, globally scoped, human-editable.
2. Memories are useful but fragile. Treat them as a staging area, not a permanent store.
3. Plans are underused across sessions. Best record of why decisions were made, but don't resurface automatically.
4. Self-correction requires forcing functions. Instructions alone aren't enough.
5. Most of ~/.claude/ is plumbing. Only 6–8 of ~15 directories are worth touching.
